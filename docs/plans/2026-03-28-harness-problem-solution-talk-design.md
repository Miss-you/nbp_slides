# Design: AI Coding Agent 分享（问题-解决型，保留时间顺序）

## 目标

- 面向 `40-80` 分钟分享，服务 `1/3 学生 + 2/3 职工` 的混合听众。
- 不做文章综述，不按公司平均分配，而是按 `问题 -> 解法` 组织内容。
- 整体保留时间顺序，让听众看到 agent 工程实践如何一步步演化。
- 强调高 ROI、可落地、可说服保守企业的经验，尤其贴近 `debug / review / eval / 约束` 场景。

## 明确取向

- **采用方案**: 问题-解决型 + 保留时间顺序。
- **不采用**: 按公司分章、按 18 篇文章平铺、只走 Anthropic 单线。
- **风格要求**: 黏土风格，少字，多图，用具象场景代替大段文字。
- **生产方式**: 两段式内容生产。
  - 先生成独立内容真相源文件。
  - 再从该文件派生 `outline_visual.md` 和最终 PPT。

## 内容主线（5+1）

### 1. Building Effective Agents

- **问题 1**: 什么时候根本不该上 agent？
- **问题 2**: 为什么一上来做“大而全 agent”往往失败？
- **解法**:
  - 从单次调用起步
  - 只在必要时升级到 workflow
  - 再用指标决定是否进入 agent

### 2. Context Engineering

- **问题 1**: 为什么上下文越多不一定越好？
- **问题 2**: 为什么 agent 会失焦、跑偏、忘重点？
- **解法**:
  - 最小高信号 token 集
  - just-in-time context
  - compaction + notes 分离

### 3. Effective Harnesses for Long-Running Agents

- **问题 1**: 长任务为什么会忘、会漂、会误报完成？
- **问题 2**: 为什么“看起来完成了”常常不等于“真的完成了”？
- **解法**:
  - 状态三件套
  - 固定 startup sequence
  - E2E / 集成验证门禁

### 4. Demystifying Evals

- **问题 1**: 怎么知道 AI 真的更好了？
- **问题 2**: 为什么“感觉更聪明”不等于“更可靠”？
- **解法**:
  - 从真实失败案例启动 eval
  - 先验证 grader，再优化 agent
  - outcome 优先于 path

### 5. Building a Better Bugbot

- **问题 1**: 企业今天最容易拿到 ROI 的 AI 用法是什么？
- **问题 2**: 怎么让 AI 的 debug / review 结果更可信？
- **解法**:
  - AI 先做 review / bug hunting / autofix 候选
  - 并行投票 + validator 提精度
  - 人工 merge 兜底

### 6. 展望：Harness Engineering

- **问题**: 如果以后 AI 更深地进入团队流程，核心变化是什么？
- **解法方向**:
  - repo 成为 system of record
  - 规则写进 linter / CI
  - 持续清理 AI slop

## 取舍原则

- 保留能直接回答“今天怎么更稳地用 AI”的材料。
- 保留能让学生建立认知台阶的材料。
- 删除容易把叙事带偏到“未来 spectacle”的材料。

## 不作为主线的材料

- `Scaling Agents`: 太容易把焦点带到数百 agent 的宏大叙事。
- `Parallel Claudes C Compiler`: 案例极端，适合作为旁证，不适合作为主轴。
- `AI-Resistant Evals`: 更偏评测设计，不是当前听众的第一优先级。
- `Codex Agent Loop` / `Codex App Server`: 过于底层，偏平台工程。

## 页数与时长策略

- **目标页数**: `24-30` 页。
- **核心版时长**: `45-50` 分钟。
- **延展版时长**: `70-80` 分钟。
- 扩展时不增加新的主线文章，只增加：
  - 每篇 1 个案例
  - 每篇 1 组“今天怎么试”的落地建议

## 页面设计原则

- 每页只讲 `1` 个核心信息。
- 文字尽量短，单页不依赖长段阅读。
- 同一内容优先画成：
  - 路径图
  - 对比图
  - 流程图
  - 仪表盘
  - 具象隐喻场景

## 执行闭环

### Phase 1: 裁剪与收束

- **规划**: 锁定 5+1 与每篇的 1-2 个问题。
- **执行**: 写入 master 文档。
- **验收**: 检查是否服务当前听众，而不是材料罗列。

### Phase 2: 大纲与演讲稿

- **规划**: 先页级结构，再演讲稿。
- **执行**: 生成 master 文档中的完整讲稿与页意图。
- **验收**: 检查叙事顺序、少字多图、单页单点。

### Phase 3: PPT 生成

- **规划**: 将 master 文档映射为 `outline_visual.md`。
- **执行**: 套用黏土风格生成图片并导出 PPTX。
- **验收**: 先抽样页，再全量生成。
